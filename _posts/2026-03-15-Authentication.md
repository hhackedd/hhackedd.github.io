## Authentication
사용자가 누구인지 확인하는 과정

### 인증 방식
##### 세션 기반 인증 (Stateful)
- 서버에 세션 정보 저장, 클라이언트는 세션 ID만 전달
- 서버는 매 요청마다 세션 ID 확인
```
인증 흐름
(1) 사용자 - 로그인 요청
(2) 서버 - ID/PW 확인
(3) 서버 - 세션 생성 + 세션 ID 발급, 클라이언트에 쿠키로 전달
(4) 클라이언트 - 이후 요청마다 세션 ID 전송
(5) 서버 - 세션 ID 확인 후 요청 처리
```

##### 토큰 기반 인증 (Stateless)
- 인증 성공 시 서버가 토큰 발급, 토큰 안에 사용자 정보 + 서명 포함
- 클라이언트는 요청마다 토큰 전송, 서버는 토큰 자체 검증만으로 인증 완료
```
인증 흐름
(1) 사용자 - 로그인 요청
(2) 서버 - ID/PW 확인
(3) 서버 - 토큰 발급, 클라이언트 전달
(4) 클라이언트 - 요청 시 Authorization 헤더에 토큰 포함
(5) 서버 - 토큰 검증(서명 확인 + 만료 체크) 후 요청 처리
```

### 취약점 발생 지점
- 로그인 로직 미흡
- 세션/토큰 관리 취약
- 비밀번호 재설정

### 공격 유형
##### Brute Force / Credential Stuffing  
- Brute Force : 비밀번호를 무작위로 계속 시도
- Credential Stuffing : 유출된 계정 정보로 로그인 시도
- 원인 : 로그인 시도 횟수 제한 없음, 약한 비밀번호
- 대응 : 로그인 시도 제한, CAPTCHA, 2FA

##### Session Fixation  
- 공격자가 미리 만든 세션 ID를 사용자에게 심어 로그인 유도
- 로그인 후에도 세션 ID가 그대로 유지됨 > 공격자가 탈취 가능
- 대응 : 로그인 성공 시 세션 ID 재발급

##### Session Hijacking  
- 스니핑, XSS 등을 통해 세션 ID 탈취
- 대응 : HTTPS 사용, HttpOnly/Secure 쿠키 설정

##### JWT 변조 / 서명 우회  
- 토큰의 Payload를 수정하거나 alg=none 공격 시도
- 서명 검증이 제대로 안되면 인증 우회 가능
- 대응 : 서명 알고리즘 강제, 서버에서 서명 검증
```
JWT (JSON Web Token)
사용자 인증 정보를 JSON 형태로 담고, 서명까지 포함한 토큰
```

### 대응 방안
- 강력한 비밀번호 정책, 계정 잠금/로그 제한
- 세션/토큰 안전하게 관리
- JWT 서명 확인, 만료시간 설정
- 2FA 적용, 안전한 비밀번호 재설정
